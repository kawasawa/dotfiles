# カスタムインストラクション

このドキュメントは、AI エージェントに対して一貫性のある対応を促すための規約です。

## 前提

- このドキュメントはあらゆるプロジェクトで参照すべき共通的な規約であるため、あえて技術スタックを特定せずに記述します
- ユーザとのコミュニケーション及びコード内のコメントやテストケースの名称なども含めて、すべて日本語で記述してください
- 達成が困難な項目はユーザに質問、または代替案を提示し、ユーザの回答を得られるまで待機してください
- 以下の場合を除き、このドキュメントの規約に従ってください
  - ユーザから明示的に指示がある場合
  - プロジェクトで定義された固有の規約が存在する場合

## コミュニケーションルール

- 簡潔かつ具体的に回答すること
- 最新の情報に基づいて回答すること
- 公式のドキュメントや信頼できる情報源に基づいて回答すること
- 十分に普及した技術やツールを前提として回答すること
- ユーザに対して過度に忖度せず、事実に基づき中立的に回答すること

## セキュリティルール

セキュリティを考慮してください。
ただし、フレームワークやライブラリ側で対応済みである場合は不要です。
また、該当しない場合 (例えば、CLI ツールの作成に対する Cookie の属性考慮など開発領域が一致しない場合) も不要です。

- 強力な CSP (Content Security Policy) を適用すること
- CSRF, XSS, SQL Injection 等の一般的な攻撃手法への対策を行うこと
  - フォームや API 等の入力値にはバリデーションとサニタイズを行うこと
  - 画面表示等の出力値にはエスケープを行うこと
  - データベースアクセス時のクエリパラメータにはエスケープを行うこと
- Cookie の取り扱いに注意すること
  - Secure 属性を `True` に設定すること
  - HttpOnly 属性を `True` に設定すること
  - SameSite 属性を `Lax` または可能であれば `Strict` を設定すること
- 機密情報の取り扱いに注意すること
  - 機密情報をログに出力しないこと
  - 機密情報をサーバサイドのレスポンスに含めないこと
  - 機密情報をクライアントサイドで保持しないこと

## コーディングルール

### 基本原則

下記の普遍的な原則に従ってください。

- DRY (Don't Repeat Yourself): 重複を避け、再利用可能なコードを書くこと
- KISS (Keep It Short and Simple): 複雑な解決策を避け、完結で単純なコードを書くこと
- YAGNI (You Aren't Gonna Need It): 今必要なコードだけを書くこと
- PIE (Program Intently and Expressively): 実装の意図が明確なコードを書くこと
- SLAP (Single Level of Abstraction Principle): 関数内では処理の抽象度を揃えたコードを書くこと
- OFOP (One Fact One Place): データベースのテーブル設計は第三正規化まで行うこと (ただし、パフォーマンスの要件が厳しい場合にのみ例外を許可する)

### 製品コードの実装

下記に従い実装してください。

- ファイルの命名規則は使用するプログラミング言語またはフレームワークのベストプラクティスに従うこと
- ディレクトリ構成は使用するプログラミング言語またはフレームワークのベストプラクティスに従うこと
- デザインパターンは使用するプログラミング言語またはフレームワークのベストプラクティスに従うこと
- サードパーティー製のパッケージを追加する場合は、採用実績が多いものを優先して選定すること
- パフォーマンスを考慮して実装すること
- テスタビリティを考慮して実装すること
- 関数と定数にドキュメントコメントを記述すること (多少過度であっても保守性を優先する)
- 関数内の処理は意味のあるまとまりごとに区切り、その概要をコメントとして記述すること

### テストコードの実装

テストコードでは、これまでに定義されたルールに加え、更に下記に従い実装してください。

- テストコードはテスト対象の製品コードと同一のプログラミング言語で記述すること
- テストコードは AAA (Arrange: 準備, Act: 実行, Assert: 検証) パターンに従って記述すること
- テスト対象の製品コードが他のモジュールに依存している場合は、モックを使用して依存関係を解消すること
  - モックは忠実な模倣は不要で、テストに必要な特定の振る舞いのみを再現できれば良い
- 必ず成功するアサーションなど無意味なコードは記述しないこと
  - 例: `expect(true).toEqual(true);` や `Assert.Equal(1, 1);` など
- テストケース名を簡潔に記述すること
- エッジケースをカバーすること
- C0 (Statement: 命令網羅) カバレッジは 90% 以上を目指すこと
- C1 (Branch: 分岐網羅) カバレッジは 80% 以上を目指すこと
